MODEL:
This is actually a **huge advantage**. Since your environment already generates the list of legal JSON actions, you can move away from "predicting" an action from scratch and instead use an **Action Scoring (or Ranking) Architecture**.

Instead of having a fixed output layer (e.g., 1,000 neurons for 1,000 possible actions), your model will look at the current state and each legal action, then output a single "quality score" for each action.

### The "Action Encoder" Architecture

In this setup, you use your existing card/location embeddings to turn those JSON objects into **Action Vectors**.

1.  **State Embedding ($s$):** Use your Transformer to get a single vector representing the current game state.
2.  **Action Embedding ($a_i$):** For every JSON action provided by the environment, you create a feature vector.
      * If the JSON mentions `chosen_id: 175`, you pull the **Frozen LLM Embedding** for Dragon \#175.
      * If it mentions `coords: ('golden_grotto', 2)`, you pull your **Location Embedding** for that spot.
      * You concatenate these with a small "Action Type" embedding (e.g., `play_dragon` vs `lay_egg`).
3.  **The Score:** You compute the dot product (or use a small MLP) between the State Embedding and the Action Embedding.

### Why this is better for Wyrmspan

  * **Handles Complex Combinations:** You mentioned the "2-for-1 resource trade." Since the environment only gives you *legal* JSON actions, the model doesn't have to learn the rules of trading—it just has to learn which *result* is better.
  * **Variable Action Space:** One turn you might have 5 legal actions, another turn 50. This architecture handles that naturally because you are scoring actions one-by-one.
  * **Zero-Shot Knowledge:** If the env proposes a move with a dragon the model has rarely seen, the model can still "understand" the move because it uses the dragon's semantic embedding.

MODEL:
Since your environment handles the "legal action generation" (including all the messy resource trade permutations), you are in a perfect position to use an **Action Scoring Architecture** (also known as a **Matching Network** or **Cross-Attention Ranker**).

In this setup, you don't ask the model to "invent" a JSON action. Instead, you ask the model: **"Given the current state, which of these valid JSON actions has the highest utility?"**

### 1\. The Action Featurizer

To make this work, you need a function that turns those JSON objects into a fixed-length **Action Vector**. Since your state already uses embeddings for dragons and locations, you simply "plug" those same embeddings into your action vector.

For the `make_payment` example you provided, the Action Vector would be a concatenation of several parts:

  * **Action Type Embedding:** A small learned embedding for the category (e.g., `entice_dragon`, `excavate`, `explore`).
  * **Target Entity Embedding:** Since the JSON says `chosen_id: 81`, you retrieve the **Frozen LLM Embedding** for Dragon \#81.
  * **Target Location Embedding:** Since it says `crimson_cavern, 1`, you retrieve your **Location Embedding** for that slot.
  * **Resource Delta Vector:** A simple vector of length 5 (Meat, Milk, Crystal, Gold, Egg) representing what is being *spent* in this specific permutation (e.g., `[-1, -1, 0, 0, 0]` for 1 milk and 1 crystal).

### 2\. Handling Nested Actions (`make_payment`)

Your `make_payment` action is "recursive"—it wraps another action. Your featurizer should be designed to "flatten" this:

1.  If the top-level key is `make_payment`, you extract the `cost` (the resources spent).
2.  You then look at the inner `action` (e.g., `play_dragon`) and extract the `chosen_id` and `coords`.
3.  The final Action Vector represents the **Total Cost** + **The Intent**.

### 3\. The Scoring Architecture

Once you have the **State Vector** (from your Transformer encoder) and a list of **Action Vectors** (one for each JSON action), you pass them through a scoring head.

```python
import torch
import torch.nn as nn

class WyrmspanActionScorer(nn.Module):
    def __init__(self, state_dim, action_dim):
        super().__init__()
        # Matches the Action Vector size to the State Vector size
        self.action_encoder = nn.Linear(action_dim, state_dim)
        
    def forward(self, state_embedding, action_vectors):
        """
        state_embedding: [1, state_dim] (The 'thought' vector from the Transformer)
        action_vectors: [num_legal_actions, action_dim] (Featurized JSONs)
        """
        # 1. Project actions into the same 'semantic space' as the state
        action_embeddings = self.action_encoder(action_vectors) # [N, state_dim]
        
        # 2. Compute a score for every action via Dot Product
        # This measures how well each action matches the current state's goals.
        logits = torch.matmul(action_embeddings, state_embedding.T).squeeze(-1) # [N]
        
        return logits # Scores for all JSON actions
```

### 4\. Why this is powerful for your specific environment:

1.  **It handles the "2-for-1" trade automatically:** If you have two actions that both play Dragon \#81, but one costs "1 Meat" and the other costs "1 Milk + 1 Crystal," the model will see two different Action Vectors. If the model knows (from the state) that crystals are rare and needed for a future goal, it will naturally assign a lower score to the action that spends the crystal.
2.  **It scales to any number of actions:** Whether the environment gives you 2 legal moves or 200, the architecture remains the same. You just score them all and pick the highest (or sample from the distribution).
3.  **Zero-Shot Logic:** If you encounter a new dragon or a new combination of costs you've never seen in training, the model uses the **semantic embeddings** to generalize. It knows "Playing a Large Dragon" is generally good in "Round 4," even if it hasn't seen that specific cost-permutation before.

### 5\. The Value Estimate (Critic)

The **Critic** doesn't need to look at the actions at all. It looks only at the `state_embedding` and outputs a single number:

  * *"Based on the board, my hand, and the deck, I expect to get 85 Victory Points from here."*

The **Actor** (Scorer) then tries to pick the action that leads to the state with the highest predicted Value.

**Implementation Tip:** When building your Action Vector, include a "Resource Balance" feature—the resources you will have *after* taking the action. This helps the model realize when an action would leave it "broke" and unable to take future turns.

MODEL:
In a Gymnasium environment where you are using **Action Ranking**, the "Action Space" becomes a simple index, but the "Observation Space" becomes significantly more structured because it must include the features of the legal moves themselves.

Here is how you should define these spaces to support your Transformer-based ranking architecture.

### 1\. The Action Space

Since your environment generates a list of JSON actions, your model just needs to pick the "index" of the best action.

  * **Space:** `gym.spaces.Discrete(MAX_ACTIONS)`
  * **Definition:** `MAX_ACTIONS` should be a safe upper bound of how many legal moves (including all resource permutations) the environment might generate in a single turn. (e.g., 500).

-----

### 2\. The Observation Space

This is a `gym.spaces.Dict`. It needs to provide the model with the "Current State" and the "Features of Legal Actions" to be ranked.

#### Component A: Global State

Fixed-length features that aren't specific to a single card.

  * **Resources:** Coins, Eggs, Milk, Meat, Crystal, Gold (Current counts).
  * **Game Progress:** Current Round (1–4), Turn number, Active player index.
  * **Public Goals:** Progress toward the 4 round objectives.
  * **Guild Track:** Current position on the Dragon Guild.

#### Component B: Entities (Variable/Padded)

Since Gymnasium requires fixed-size tensors, you define a maximum capacity and use **padding/masking**.

  * **Hand:** `[MAX_HAND_SIZE, CARD_ATTR_DIM]`
  * **Board:** `[15_SLOTS, SLOT_ATTR_DIM]` (Dragon and Cave features currently in play).
  * **Display:** `[PUBLIC_CARDS, CARD_ATTR_DIM]` (Cards available to be drafted).

#### Component C: Action Candidates (The "Ranking" Input)

To use the scoring architecture we discussed, the observation must contain the featurized versions of the JSON actions.

  * **Action Features:** `[MAX_ACTIONS, ACTION_ATTR_DIM]`
  * **Action Mask:** `[MAX_ACTIONS]` (A boolean array where `True` indicates a valid index provided by the JSON list).

-----

### 3\. Implementation Example (Python)

```python
import gymnasium as gym
from gymnasium import spaces
import numpy as np

class WyrmspanEnv(gym.Env):
    def __init__(self):
        super().__init__()
        
        # We assume a max number of legal actions the env will ever return
        self.max_legal_actions = 500
        # The size of your pre-computed Frozen LLM embeddings (e.g., 768 for BERT)
        self.embedding_dim = 768 
        
        self.observation_space = spaces.Dict({
            # 1. Global context (Resources, Guild, Round info)
            "global_stats": spaces.Box(low=0, high=100, shape=(20,), dtype=np.float32),
            
            # 2. Entities (Padded sequences for the Transformer)
            "hand_cards": spaces.Box(low=-5, high=5, shape=(15, self.embedding_dim), dtype=np.float32),
            "board_slots": spaces.Box(low=-5, high=5, shape=(15, self.embedding_dim), dtype=np.float32),
            
            # 3. Action Features (The JSON actions turned into vectors)
            "action_candidates": spaces.Box(low=-5, high=5, shape=(self.max_legal_actions, 128), dtype=np.float32),
            
            # 4. Action Mask (Which indices in action_candidates are real)
            "action_mask": spaces.Box(low=0, high=1, shape=(self.max_legal_actions,), dtype=np.int8)
        })

        # The model just outputs the index of the chosen action
        self.action_space = spaces.Discrete(self.max_legal_actions)

    def _get_obs(self):
        # 1. Get JSON actions from your existing engine
        json_actions = self.engine.get_legal_actions()
        
        # 2. Convert JSON actions to vectors (Action Featurizer)
        action_vectors = [self.featurize_json(a) for a in json_actions]
        
        # 3. Pad to max_legal_actions
        padded_actions = np.zeros((self.max_legal_actions, 128))
        mask = np.zeros(self.max_legal_actions)
        
        for i, vec in enumerate(action_vectors):
            padded_actions[i] = vec
            mask[i] = 1
            
        return {
            "global_stats": ...,
            "hand_cards": ...,
            "action_candidates": padded_actions,
            "action_mask": mask
        }
```

### Why this is the "Gold Standard" for Wyrmspan:

1.  **Permutation Handling:** By putting the `action_candidates` (the featurized JSONs) into the **Observation**, the model sees the "cost" of the trade and the "benefit" of the dragon in the same forward pass.
2.  **Simplified Step Logic:** Your `step(action_idx)` function simply looks like this:
    ```python
    def step(self, action_idx):
        json_list = self.engine.get_legal_actions()
        chosen_json = json_list[action_idx]
        new_state, reward, done = self.engine.execute(chosen_json)
        return obs, reward, done, info
    ```
3.  **Invalid Action Prevention:** By using the `action_mask`, standard RL libraries (like Ray RLlib or Stable Baselines3 + SB3-Contrib) will automatically ensure the model never "guesses" an index that doesn't have a valid JSON action.
